    # Map实现类

    # 一、HashMap

    # 1、特征

    • 拥有map的一切特征：键值对、无序不可重复...

    • 默认初始化容量16，加载因子：0.75（即当集合/数组容量达到75%时自动开始扩容）

    • 自动扩容后的容量是原容量的2倍

    • 重点记住：HashMap集合的初始化容量必须是2的倍数（为了达到散列均匀，以提高哈希表的存取效率）

    • 底层是哈希表/散列表数据结构

    • 该哈希表由一个数组和一个单项链表组成，即拉链存储法（数组中每一个元素是一个单向链表）

    • HashMap底层部分源代码：

      public class HashMap(){
    
    //HashMap底层实际上就是个一维数组
    transient Node<K,V>[] table;
    
    //静态内部类HashMap.Node（单向链表）
    static class Node<K,V>{
        //哈希值：哈希值是key通过hashCode()方法得出的结果。hash值通过哈希函数计算，可以转换存储成数组的下标
        final int hash;
        final K key;
        V value;
        Node<K,V> next;	//指向下一个结点的内存地址
    }
    
}

    # 2、put(k,v)方法的实现原理

    1、先将key，value封装到Node对象当中

    2、底层调用key的hashCode()方法得出hash值，然后通过哈希函数/哈希算法，将hash值转换成数组的下标。

    3、下标位置上如果没有任何元素，就把Node添加到这个位置上。

    4、如果下标对应的位置上有链表，此时会拿着key和链表上每一个节点的key进行equals比较，如果所有的equals()方法返回都是false，那么这个新节点将被添加到链表的末尾。如果其中有一个equals返回true，那么这个节点的value将会被覆盖

    get()方法同理

    # 小结：

    1. 哈希表的随机增删和查询效率都不低：增删在链表上，查询主要在数组上
    2. HashMap集合key特点：
      • 无序：因为输入时不知道会挂到哪个链表上，取出时就乱了
      • 不可重复：依靠equals()保证。如果key重复了，value会覆盖
    3. HashMap集合的key会先后调用两个方法：hashCode()和equals()
    4. 以上两个方法需要重写
    5. 同一个单向链表上的元素（node对象）存的hash可能一样，但key肯定都不一样。（虽然hash值可能不同，但hash值经过哈希函数计算后得出的下标一定是相同的，这种情况称为"哈希碰撞"或”冲突“）
    6. hashCode()重写不当时会导致数组频繁扩容（每个单链表只有一两个节点）或查询效率降低（只剩下几条很长的单链表）

    # 3、重写hashCode()和equals()两个方法

    //假设s1和s2内容相同
Student s1 = new Student("zhangsan");
Student s2 = new Student("zhangsan");

//equals()方法
System.out.println(s1.equals("zhangsan"));	//不重写方法为false

//若不重写hashCode，会调用地址进行哈希运算，以下两个结果不同
s1.hashCode();
s2.hashCode();

//若只重写了equals()，却没有重新hashCode()
Set<Student> students = new HashSet<>();
students.add(s1);
students.add(s2);
System.out.println(students.size());	//结果按说应该是1，但结果是2
										//不符合set集合的要求（不可重复）

    注意：

    • HashMap集合的key会先后调用两个方法：hashCode()和equals()

    • 以上两个方法需要重写

    • 放在HashMap集合中的key元素实际上是放到HashSet集合中，因此HashSet集合中的元素也需要重写hashCode()和equals()两个方法

    • 向Map集合中存，以及从Map中取，都是先调用key的hashCode()方法，然后再调用equals()方法。而且equals()方法有可能调用，也有可能不调用

      • 分析（以put(key,value)方法举例）：

        1、key.hashCode()方法返回哈希值

        2、哈希值经过哈希算法转换成数组下标

        3、数组下标位置如果是null，则equals()不执行

    • 如果一个类的equals()方法重写了，那么hashCode()方法必须重写。并且equals()方法返回如果是true，hashCode()方法返回的值必须一样

      • 分析：

        equals方法返回true表示两个对象相同，在同一个单向链表上进行比较

        那么对于同一个单向链表上的节点来说，他们的哈希值都是相同的（哈希碰撞/冲突情况除外）

        所以hashCode()方法的返回值也应该相同

    • hashCode() 方法和equals()方法不用研究，直接使用IDEA工具生成，但是这两个方法需要同时生成

    终极结论：

    放在HashMap集合key部分，以及放在HashSet集合中的元素，需要同时重新hashCode方法和equals方法

    # 4、JDK8新特性

    从JDK8开始，HashMap加入红黑树数据结构，以提高检索效率

    如果哈希表单向链表中元素超过8个，单向链表会转换成红黑树；

    当红黑树上的节点数量小于6时，会重新把红黑树变成单向链表。

    # 二、Hashtable（了解）

    特征：

    1. Hashtable方法都带有synchronized关键字，表示是线程安全的
    2. 底层是哈希表
    3. 初始化容量：11；加载因子：0.75
    4. 自动扩容：原容量的两倍再+1
    5. Hashtable对线程处理效率较低，同时线程安全有其他的方案，导致使用较少了
    6. Hashtable的key和value值都不允许为null

    # 三、Properties

    特征：

    1. Properties是一个Map集合，继承Hashtable
    2. Properties是线程安全的
    3. Properties的key和value都是String类型
    4. Properties被称为属性类对象

    常用方法（两个，一个存一个取）：

    Object setProperty(key,value)

    保存字符串类型的键值对

    # 自定义排序方法一：实现compareTo()方法（实现java.lang.Comparable接口）

    自定义规则：

    • 放在TreeSet集合中的元素需要实现java.lang.Comparable接口
    • 实现compareTo()方法，equals()方法可以不重写（底层调用compareTo()方法进行比较）
    • 需要在compareTo()方法中编写比较的逻辑/规则，按照什么进行比较

    # 底层/源码分析：

    1. TreeMap底层是二叉树，且遵循左小右大的顺序进行数据存放
    2. 每添加一次元素（如persons.add(p1)），会调用TreeMap的put()方法
    3. 如果没有定义比较器，则采用以下方法：
    4. （关键！！！）put()方法会调用p1的compareTo()方法与集合（平衡二叉树）中的每个节点进行比较
    5. 若p1.compareTo(t.key)返回值>0，即要插入的p1大于二叉树上的当前节点，会往节点的右子树上找（t=t.right）
      • 返回值<0则往左子树上找，=0则进行值覆盖
      • 遍历到null时把p1插入二叉树，构成新的二叉树
    6. 因此compareTo()的返回值很重要，决定了二叉树的结构
    7. 从集合中取出元素是按照中序遍历的方式读取

    实现举例：按照年龄排序

    //实现Comparable接口
class Customer implements Comparable<Customer>{
    int age;

    //构造函数
    public Customer(int age) {
        this.age = age;
    }

    //重写toString
    @Override
    public String toString() {
        return "Customer{" + "age=" + age + '}';
    }

    //自定义规则：实现compareTo()
    //c2是要插入的元素，c1是二叉树上的节点
    @Override
    public int compareTo(Customer cus) {  //底层调用：c2.compareTo(c1);
        //this是c2
        //cus是c1
        //c2和c1比较的时候，就是this和cus比较

        //升序(记忆：前小后大)
        return this.age - cus.age;  //返回<0  >0  =0

        //降序
        //return cus.age - this.age;
    }
}

    # 自定义排序方法二：使用比较器的方法（实现java.util.Comparator接口）

    import java.util.Comparator;
import java.util.TreeSet;

public class TreeMapTest2 {
    public static void main(String[] args) {
        //创建TreeSet集合的时候，需要使用这个比较器
        //给构造方法传递一个比较器
        TreeSet<WuGui> wuGuis = new TreeSet<>(new WuGuiComparator());

        wuGuis.add(new WuGui(1000));
        wuGuis.add(new WuGui(103));
        wuGuis.add(new WuGui(600));

        for(WuGui wuGui : wuGuis){
            System.out.println(wuGui);
        }
    }
}

class WuGui{
    int age;

    public WuGui(int age) {
        this.age = age;
    }

    @Override
    public String toString() {
        return "WuGui{" +
                "age=" + age +
                '}';
    }
}

//单独在这里编写一个比较器
//比较器实现java.util.Comparator接口
class WuGuiComparator implements Comparator<WuGui> {

    @Override
    public int compare(WuGui o1, WuGui o2) {
        //指定比较规则
        //按照年龄排序
        return o1.age - o2.age;
    }
}

    # 注意：

    对于集合中的元素是String、Integer等系统已经写好比较规则（CompareTo()），无法通过重写该方法来改变排序规则（例如改为降序排序），因为源码中方法已经用final修饰

    此时可以通过传递比较器的方法来改变排序规则

    # 结论：

    # 放到TreeSet或者TreeMap集合key部分的元素想要做到排序，包括两种方式：
    1. 放在集合中的元素实现java.lang.Comparable接口
    2. 在构造TreeSet或者TreeMap集合的时候给它传一个比较器对象
    # Comparable和Comparator如何选择？
    • 当比较规则不会发生改变的时候，，建议在类里实现Comparable接口
    • 如果比较规则有多个，并且需要多个比较规则之间频繁切换，建议使用Comparator接口

    （Comparator接口的设计符合OCP原则）
